---
title: "Transactions"
description: "Transaction models and types used in the Clementine protocol"
---

The Clementine protocol uses various transaction types for different protocol operations including deposits, withdrawals, challenges, and reimbursements.

## Transaction Identifiers

### GrpcTransactionId

Unified transaction identifier supporting both normal and numbered transaction types.

<ParamField body="normal_transaction" type="NormalTransactionId">
  Normal transaction type (used once per deposit)
</ParamField>

<ParamField body="numbered_transaction" type="NumberedTransactionId">
  Numbered transaction type (used multiple times)
  
  <Expandable title="NumberedTransactionId fields">
    <ResponseField name="transaction_type" type="NumberedTransactionType" required>
      Type of numbered transaction
    </ResponseField>
    <ResponseField name="index" type="int32" required>
      Index of the transaction instance
    </ResponseField>
  </Expandable>
</ParamField>

---

</ResponseField>

<ResponseField name="UNSPENT_KICKOFF" type="enum" default="5">

## Transaction Models

### TransactionRequest

Request to create transactions for a specific deposit and kickoff.

<ParamField body="deposit_outpoint" type="Outpoint" required>
  Deposit UTXO outpoint
  
  <Expandable title="Outpoint fields">
    <ResponseField name="txid" type="Txid" required>
      Transaction ID (bytes)
    </ResponseField>
    <ResponseField name="vout" type="uint32" required>
      Output index
    </ResponseField>
  </Expandable>
</ParamField>

<ParamField body="kickoff_id" type="KickoffId" required>
  Kickoff identifier
  
  <Expandable title="KickoffId fields">
    <ResponseField name="operator_xonly_pk" type="bytes" required>
      Operator's x-only public key
    </ResponseField>
    <ResponseField name="round_idx" type="uint32" required>
      Round index in operator's collateral chain
    </ResponseField>
    <ResponseField name="kickoff_idx" type="uint32" required>
      Kickoff index within the round
    </ResponseField>
  </Expandable>
</ParamField>

## Withdrawal Models

### WithdrawParams

Parameters for withdrawal/payout transactions.

<ParamField body="withdrawal_id" type="uint32" required>
  Withdrawal ID in Citrea
</ParamField>

<ParamField body="input_signature" type="bytes" required>
  User's `TapSighashType::SinglePlusAnyoneCanPay` signature
</ParamField>

<ParamField body="input_outpoint" type="Outpoint" required>
  User's UTXO to claim the deposit
</ParamField>

<ParamField body="output_script_pubkey" type="bytes" required>
  Withdrawal output's script pubkey (signature is only valid for this pubkey)
</ParamField>

<ParamField body="output_amount" type="uint64" required>
  Withdrawal output's amount in satoshis (signature is only valid for this amount)
</ParamField>

---

### WithdrawParamsWithSig

Withdrawal parameters with optional verification signature.

<ParamField body="withdrawal" type="WithdrawParams" required>
  Core withdrawal parameters
</ParamField>

<ParamField body="verification_signature" type="string">
  Optional ECDSA signature from Citrea/aggregator for authentication
</ParamField>

---

### FinalizedPayoutParams

Parameters for finalized payout transactions.

<ParamField body="payout_blockhash" type="bytes" required>
  Bitcoin blockhash for the payout
</ParamField>

<ParamField body="deposit_outpoint" type="Outpoint" required>
  Associated deposit outpoint
</ParamField>

## Example Usage

### Transaction Request

```json
{
  "deposit_outpoint": {
    "txid": {"txid": "<32_bytes>"},
    "vout": 0
  },
  "kickoff_id": {
    "operator_xonly_pk": "<32_bytes>",
    "round_idx": 1,
    "kickoff_idx": 0
  }
}
```

### Signed Transaction with Type

```json
{
  "transaction_type": {
    "normal_transaction": "MOVE_TO_VAULT"
  },
  "raw_tx": "<raw_bitcoin_tx_bytes>"
}
```

### Numbered Transaction

```json
{
  "transaction_type": {
    "numbered_transaction": {
      "transaction_type": "MINI_ASSERT",
      "index": 42
    }
  },
  "raw_tx": "<raw_bitcoin_tx_bytes>"
}
```
